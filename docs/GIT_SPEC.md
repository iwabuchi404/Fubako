# Git連携仕様書

**バージョン:** 1.1.0
**作成日:** 2026-06-15
**目的:** 中小企業での小規模チーム運用に対応したGitバージョン管理機能

---

## 1. 概要

FubakoのコンテンツをGitでバージョン管理し、チームメンバーと共有する機能。

### 対象ユーザー
- 中小企業のWeb制作チーム
- 非エンジニアのコンテンツ担当者
- エンジニア1〜3名程度の小規模チーム

### 基本方針
- **シンプルさ**: Gitの複雑さを隠蔽し、必要な操作のみを提供
- **ファイル完全書き換え原則**: コンテンツ編集はファイル全体の書き換えを前提。3-wayマージ（行レベル自動結合）は使用しない
- **選択式同期**: コンフリクト発生時はファイル単位で「ローカルを残す/リモートを残す」を選択
- **自動化**: 定期フェッチ（3分）により、リモートの変更を検知

---

## 2. ブランチ戦略

| ブランチ | 用途 | デフォルト名 |
|:---|:---|:---|
| **develop** | 開発用ブランチ。コンテンツ編集・保存はここへ | `develop` |
| **production** | 本番用ブランチ。CI/CDでデプロイ | `main` or `production` |

### ブランチフロー
```
develop (編集・保存) ───┐
                         ├─→ production (本番公開)
                         │
                      (CI/CDでデプロイ)
```

**重要**: production ブランチは develop → production のマージ結果のみで更新される。直接のcommitやforce pushは禁止。

## 3. 技術仕様

### 3.1 ライブラリ
- **dugite**: Gitコマンド実行（ElectronからGit操作）
- **Git Credential Manager (GCM)**: 認証（OS標準インストール済みと想定）

### 3.2 ファイル完全書き換え原則

Fubakoのコンテンツ編集（Markdown記事等）は、1回の編集でファイル全体が完全に書き換わることを前提としている。

**前提条件**:
- ファイル単位で変更が完結する
- 複数人で同一ファイルを同時編集しない運用（ロックフリー）
- 3-wayマージによる行レベルの自動結合は不採用

**コンフリクト解決方針**:
- 同一ファイルで両側に変更がある場合は、自動マージを行わない
- ユーザーに「ローカル版を残す」「リモート版を残す」の選択をさせる
- 別々のファイルの変更は安全にマージ取り込み

### 3.3 Git操作フロー

#### 3.3.1 保存（Git保存ボタン）
1. `git add .` - 全変更をステージング
2. `git commit -m "Update content"` - コミット
3. `git push origin develop` - developブランチへプッシュ

**結果**: コンテンツがリモートリポジトリへ反映

#### 3.3.2 同期（更新ボタン）
**トリガー**: ダッシュボードの「更新ボタン」（手動）

**同期ロジック**:
1. `git fetch` - リモートの最新を取得
2. 分岐判定を実行
   - ローカルに未pushコミットがない（fast-forward可能）→ 自動的に `git pull` で取り込み
   - ローカルにもコミットがある場合:
     - **同一ファイルを両側で変更** → 自動マージしない。コンフリクトとして検出
     - **別々のファイル** → 安全にマージ取り込み

**コンフリクト解決UI**:
- ファイル単位で「ローカルを残す」「リモートを残す」を選択するパネルを表示
- `git checkout --ours/--theirs` で選択を適用
- 選択完了後、自動的にコミットして同期完了

**結果**: ローカルの状態がリモートと同期

#### 3.3.3 公開（本番公開ボタン）
**操作**: develop を production へ merge して push のみ

**フロー**:
1. `git fetch` - 最新を取得
2. `git checkout production` - productionブランチへ移動
3. `git merge develop` - developをproductionへマージ（通常のmerge）
4. `git push origin production` - productionへプッシュ
5. `git checkout develop` - 元ブランチへ復帰

**結果**: productionブランチが更新され、CI/CDで本番環境へデプロイ

**コンフリクト発生時の挙動**:
- production への直接操作は禁止されているため、コンフリクトは起きない前提
- 万一コンフリクトが発生した場合は、エラーを表示して公開を中止

**CI/CD連携**:
- push トリガーで CI が発火
- 本番公開が自動的に実行される

#### 3.3.4 定期フェッチ
- **頻度**: 3分に1回
- **通知**: なし（サイレント）
- **操作**: `git fetch`のみ（マージはしない）
- **目的**: リモートの変更を検知し、UIに「更新可能」状態を表示

**今後の課題**:
- 現状は fetch-only でユーザーが手動で更新ボタンを押す必要がある
- 問題が発生した場合、定期 pull（自動同期）の導入を検討

## 4. UI仕様

### 4.1 Git設定画面 (`/git-settings`)

#### 設定項目
| 項目 | デフォルト | 説明 |
|:---|:---|:---|
| Git管理 | オフ | Git機能の有効/無効を切り替え |
| 開発ブランチ名 | `develop` | 編集・保存用ブランチ |
| 本番ブランチ名 | `main` | 公開用ブランチ |
| リモートURL | - | GitリポジトリURL（例: `https://github.com/user/repo.git`） |

#### ステータス表示（Git有効時のみ）
- リポジトリ状態: 有効 / 無効
- 現在のブランチ: `develop` etc.
- 未コミットの変更: あり / なし
- リモートとの同期: 最新 / 古い / 追い越している
- 未ステージングの変更ファイル一覧

#### 操作ボタン
- 「ステータスを更新」: 最新のGit状態を取得

---

### 4.2 ダッシュボード（`/`）

#### Git管理有効時
以下のボタンを表示:

| ボタン | アクション | 確認 |
|:---|:---|:---|
| **Git保存** | commit & push to develop | なし |
| **同期（更新）** | pull（リモートを取り込み） | なし |
| **エクスポート** | dist/をzip化 | なし |
| **本番公開** | develop → production merge & push | あり |

**同期ボタンの挙動**:
- fetch → 分岐判定 → fast-forwardなら自動取り込み
- コンフリクトがあれば解決パネルを表示

**本番公開ボタンの確認ダイアログ**:
```
本番環境へ公開しますか？
よろしければ[公開]を押してください
[キャンセル] [公開]
```

#### コンフリクト解決パネル
同期中にコンフリクトが検出された場合に表示:

**UI構成**:
- コンフリクトファイル一覧
- ファイルごとの選択肢:
  - 「ローカル版を残す」ボタン（`--ours`）
  - 「リモート版を残す」ボタン（`--theirs`）
- 「適用して同期完了」ボタン

**フロー**:
1. ユーザーが各ファイルを選択
2. 「適用して同期完了」を押す
3. `git checkout --ours/--theirs` を実行
4. `git commit` でマージ完了
5. ダッシュボードへ戻る

#### Git管理無効時
- Git保存、同期、本番公開ボタンは非表示
- エクスポートボタンのみ表示

**デザイン**:

## 6. エラーハンドリング

### 6.1 Git操作エラー

| エラー | 対応 |
|:---|:---|
| `not a git repository` | Git無効として扱う |
| `nothing to commit` | 成功として扱う（変更なし） |
| `authentication failed` | エラーメッセージを表示（GCM再実行を促す） |
| `connection refused` | ネットワークエラーとして表示 |
| `conflict` | コンフリクト解決パネルを表示 |
| `force-push rejected` | production 公開時のリジェクトをエラー表示 |

### 6.2 コミットエラーの特別扱い

**`git commit` がexit code 1で失敗した場合**:
1. stdout/stderrを確認
2. 「nothing to commit」または「no changes added to commit」が含まれる場合
   - 成功として扱う（変更がないだけ）
3. それ以外のエラー
   - エラーメッセージを表示

## 7. 定期フェッチ処理

### 7.1 実装

```javascript
const FETCH_INTERVAL = 3 * 60 * 1000 // 3分

function startAutoFetch(projectPath) {
  return setInterval(() => {
    gitExec(['fetch'], projectPath)
  }, FETCH_INTERVAL)
}

// プロジェクトを開いた時に開始
// プロジェクトを閉じた時に停止
```

### 7.2 注意点

- エラーが発生しても無視（サイレント）
- ログには出力してデバッグ用に
- UIへの通知は一切しない
- マージはしない（fetchのみ）

## 8. エクスポート機能

### 8.1 動作

1. `dist/` ディレクトリを確認
2. `dist-export.zip` をプロジェクトルートに作成
3. `dist/` 内のファイルをすべてzipに含める

### 8.2 ユーザー体験

- エクスポートボタンを押すと自動的にzip作成
- 完了時にファイル保存ダイアログを表示
- 保存場所を選択してダウンロード

## 9. 動作モード

### 9.1 Git管理有効モード

**利用可能な機能**:
- Git保存ボタン
- 同期（更新）ボタン
- 本番公開ボタン
- エクスポートボタン
- Git設定画面
- コンフリクト解決パネル

**保存フロー**:
1. コンテンツ編集
2. 保存ボタン（ローカルMarkdown保存）
3. Git保存ボタン（Git commit & push）

**同期フロー**:
1. チームメンバーが変更をpush
2. 定期フェッチが変更を検知
3. ユーザーが「同期」ボタンを押す
4. コンフリクトがあれば解決パネルで選択
5. 同期完了

---

### 9.2 Git管理無効モード

- エクスポートボタンのみ

**保存フロー**:
1. コンテンツ編集
2. 保存ボタン（ローカルMarkdown保存のみ）

**公開方法**:
- エクスポートでzipをダウンロード
- 手動でデプロイ

## 11. ユーザーストーリー

### ストーリー1: 初期設定
1. ユーザーはGit設定画面を開く
2. 「Git管理」をオンにする
3. リモートURLを入力する
4. 初期化ボタンを押す（必要に応じて）
5. Git有効完了

### ストーリー2: コンテンツ編集と保存
1. ユーザーは記事を編集する
2. 保存ボタンを押す（ローカル保存）
3. Git保存ボタンを押す（Git commit & push）
4. チームメンバーが同期ボタンを押して反映

### ストーリー3: チームでの同期（コンフリクトなし）
1. ユーザーAが記事Aを編集して保存
2. ユーザーBが記事Bを編集して保存
3. ユーザーAが「同期」ボタンを押す
4. 別々のファイルなので自動的にマージ完了

### ストーリー4: チームでの同期（コンフリクトあり）
1. ユーザーAとユーザーBが同じ記事を編集
2. 双方がpushした状態で、ユーザーAが「同期」ボタンを押す
3. コンフリクト解決パネルが表示される
4. ユーザーAは「ローカル版を残す」を選択
5. 同期完了

### ストーリー5: 本番公開
1. ユーザーは記事の編集が完了したと判断する
2. 本番公開ボタンを押す
3. 確認ダイアログで「公開」を押す
4. develop→productionにマージ
5. CI/CDが自動的に本番環境へデプロイ
